# Відповіді на контрольні запитання (Практична робота №3)

1. **Чому використання if/else всередині JSX викликає помилку?**
JSX — це синтаксичний цукор для викликів `React.createElement()`, що є виразами (expressions). Конструкції `if/else` є інструкціями (statements) і не повертають значення безпосередньо, тому їх не можна вставляти всередину фігурних дужок JSX. Замість них використовують тернарні оператори або логічне `&&`.

2. **В чому небезпека числа 0 у лівій частині оператора &&?**
У JavaScript число `0` є хибним значенням (falsy), але React рендерить `0` як текст. Якщо написати `{list.length && <Component />}`, і `list.length` дорівнює `0`, React виведе символ "0" на екран замість того, щоб нічого не показувати. Краще використовувати `{list.length > 0 && <Component />}`.

3. **Як реалізувати рендеринг для 5+ варіантів?**
Для великої кількості варіантів найкраще використовувати:
- Об'єктний мапінг: `const components = { a: <A />, b: <B />, ... }; {components[activeTab]}`
- Конструкцію `switch/case` у окремій функції або фрагменті коду перед `return`.

4. **Коли тернарний оператор кращий за &&?**
Тернарний оператор кращий, коли потрібно реалізувати логіку "якщо інакше" (if-else), тобто показати один компонент, якщо умова істинна, і інший — якщо хибна. Оператор `&&` підходить лише для логіки "якщо — то" (if-only).

5. **Як React розуміє, що треба перерендерити інтерфейс при зміні вкладки?**
React відстежує стан через хук `useState`. Коли ми викликаємо функцію оновлення стану (наприклад, `setActiveTab`), React фіксує зміну стану, позначає компонент як "брудний" (dirty) і ініціює процес звірки (reconciliation) віртуального DOM, що призводить до оновлення реального DOM.
