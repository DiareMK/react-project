## 1. Фрагмент коду з логікою пошуку та фільтрації

Логіка об'єднує в собі і пошук за текстом/автором, і фільтрацію за обраною категорією:

```javascript
const filteredPosts = postsData.filter(post => {
  // Логіка пошуку (insensitive до регістру)
  const matchesSearch = 
    post.content.toLowerCase().includes(searchTerm.toLowerCase()) ||
    post.author.toLowerCase().includes(searchTerm.toLowerCase());
  
  // Логіка категорій
  const matchesCategory = activeCategory === 'All' || post.category === activeCategory;

  return matchesSearch && matchesCategory;
});
```

## 2. Пояснення реалізації "порожнього результату"

Для відображення повідомлення про відсутність результатів використовується умовний рендеринг за допомогою тернарного оператора, який перевіряє довжину відфільтрованого масиву:

```javascript
{filteredPosts.length > 0 ? (

```

## 3. Відповіді на контрольні запитання

**Що таке "підняття стану" і чому ми використовуємо його для SearchBar?**
"Підняття стану" (Lifting State Up) — це патерн у React, коли стан переноситься до найближчого спільного предка компонентів, яким цей стан потрібен. Для `SearchBar` ми піднімаємо стан в `App.jsx`, щоб батьківський компонент міг використовувати значення пошуку для фільтрації списку постів, які знаходяться поза `SearchBar`.

**Пояснення асинхронної природи оновлення стану в useState.**
Оновлення стану через `setSearchTerm` або `setActiveCategory` не відбувається миттєво. React групує оновлення для оптимізації продуктивності (batching) і планує перемальовування компонента. Значення стану зміниться лише під час наступного рендеру.

**Чому для фільтрації створюється нова змінна, а не змінюється оригінальний масив у стані?**
В React стан має бути незмінним (immutable). Пряма мутація `postsData` призвела б до втрати оригінальних даних і потенційних багів з рендерингом. Створення нової змінної `filteredPosts` через `.filter()` дозволяє нам обчислювати потрібні дані "на льоту" під час рендеру, зберігаючи оригінальний масив цілісним.

**В чому перевага керованих компонентів над некерованими при пошуку?**
Керовані компоненти (через `value` та `onChange`) дають повний контроль над введенням. Це дозволяє:
- Реалізувати миттєву фільтрацію при кожному натисканні клавіші.
- Легко очищувати поле пошуку програмно.
- Валідувати або форматувати ввід користувача в режимі реального часу.
- Синхронізувати пошуковий запит з іншими елементами інтерфейсу.
