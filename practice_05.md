## 1. Код компонентів картки товару

Для реалізації картки товару використано 3 рівні ієрархії:
- **Контейнер** (Smart): `ProductContainer`
- **Деталі та Дії** (Dumb/Compositional): `ProductDetails`, `ProductActions`
- **Атомарні компоненти** (Dumb): `StarRating`, `Button`

### ProductContainer.jsx
```jsx
const ProductContainer = () => {
  const [quantity, setQuantity] = useState(1);
  const productData = { name: '...', description: '...', price: 12500, rating: 5 };

  const handleAdd = () => setQuantity(prev => prev + 1);
  const handleRemove = () => setQuantity(prev => (prev > 0 ? prev - 1 : 0));

  return (
    <div className="card">
      <ProductDetails {...productData} />
      <ProductActions 
        quantity={quantity} 
        onAdd={handleAdd} 
        onRemove={handleRemove} 
      />
    </div>
  );
};
```

## 2. Відповіді на контрольні запитання

**Які відмінності між Smart та Dumb компонентами?**
- **Smart (Контейнери):** Відповідають за логіку, роботу зі станом та отримання даних (API). Вони "знають", як працює додаток.
- **Dumb (Презентаційні):** Відповідають тільки за зовнішній вигляд. Вони отримують дані через props і не мають власної бізнес-логіки. Це робить їх легкими для тестування та повторного використання.

**Пояснення принципу "Джерело істини" (Single Source of Truth)**
У React це означає, що певна частина стану має належати лише одному компоненту. Всі інші компоненти, яким потрібен цей стан, отримують його через props. Це запобігає розсинхронізації даних та робить поведінку додатку передбачуваною.

**Чому map знаходиться в композиційних компонентах?**
Композиційні компоненти (як `ActivityList` або `ProductList`) призначені для структурування групи елементів. Використання `map` саме тут дозволяє тримати відображення списку в одному місці, відокремлюючи логіку отримання масиву від логіки рендерингу окремого елемента.

**Що таке Prop Drilling і коли він стає проблемою?**
Prop Drilling - це процес передачі даних через кілька рівнів вкладених компонентів, які самі ці дані не використовують, а лише "прокидають" їх далі. Це стає проблемою, коли рівнів вкладеності стає занадто багато (4-5+), оскільки код стає важким для підтримки та рефакторингу.

**Критерії для винесення розмітки в окремий компонент:**
1. **Повторне використання:** Якщо шматок розмітки потрібен у кількох місцях.
2. **Складність:** Якщо компонент стає занадто великим (більше 100-200 рядків) і його важко читати.
3. **Логічна відокремленість:** Якщо частина коду виконує одну конкретну, завершену функцію (наприклад, кнопка з іконкою або зірковий рейтинг).
